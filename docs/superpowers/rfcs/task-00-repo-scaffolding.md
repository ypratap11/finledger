# Task 0 RFC — Repository Scaffolding

**Plan reference:** `docs/superpowers/plans/2026-04-14-finledger-m1.md` Task 0.

**What this task delivers:** A runnable, empty-but-correct repo skeleton. After it, you can `docker compose up postgres` and have a healthy DB. Nothing more.

**Why this task exists:** Every later task assumes (a) a Postgres reachable at a known URL, (b) a `.env.example` documenting the env vars each component needs, (c) a `.gitignore` that keeps secrets and build artifacts out of git. Doing all three together in one commit gives the repo a clean "zeroth state" before any real code.

---

## File 1: `.gitignore`

### What it contains
```
node_modules/
dist/
__pycache__/
*.pyc
.pytest_cache/
.ruff_cache/
.mypy_cache/
.venv/
venv/
.env
*.egg-info/
coverage/
.coverage
```

### Why each line

| Line | Reason |
|------|--------|
| `node_modules/` | npm's output dir; hundreds of MB, regenerated by `npm install`. Committing it bloats the repo and causes merge conflicts on every dep change. |
| `dist/` | TypeScript's compiled output; generated by `tsc` from `src/`. Source is the truth; compiled artifacts should never be committed. |
| `__pycache__/` and `*.pyc` | Python's bytecode cache. Regenerated on first import. Committing them causes stale-cache bugs when Python version changes. |
| `.pytest_cache/` | pytest's per-repo cache of last-run info. Machine-local; leaking someone else's cache into your checkout causes false reruns. |
| `.ruff_cache/`, `.mypy_cache/` | Same reasoning: lint/typecheck caches. |
| `.venv/`, `venv/` | Virtualenv directories. Environment is per-developer; `pyproject.toml` declares dependencies, the venv is where they land locally. |
| `.env` | **Critical for security.** Developer-local env file with real secrets. Committing `.env` is the #1 way real projects leak API keys to public repos. We commit `.env.example` instead (placeholder values only). |
| `*.egg-info/` | Python packaging metadata, generated by `pip install -e`. |
| `coverage/`, `.coverage` | Coverage reports. Results of a test run, not inputs to one. |

### Alternatives considered
- **One big `.gitignore` with every tool's patterns (GitHub's template).** Rejected: we don't use most of those tools. A minimal `.gitignore` documents what the project actually uses.
- **Separate `.gitignore` per subdirectory.** Rejected for M1: we have two subprojects (`core/`, `ingest-edge/`), but the global set above already covers both. Subdir-specific rules can be added later if a subproject needs something global shouldn't include.

## File 2: `.env.example`

### What it contains
```
DATABASE_URL=postgresql://finledger:finledger@localhost:5432/finledger
STRIPE_WEBHOOK_SECRET=whsec_test_placeholder
STRIPE_API_KEY=sk_test_placeholder
ZUORA_WEBHOOK_SECRET=zuora_test_placeholder
INGEST_PORT=3001
CORE_PORT=8000
```

### Why each line

| Var | Reason |
|-----|--------|
| `DATABASE_URL` | Single Postgres connection string. Both the Python core and the Node edge read this same URL, so both services share one source of truth for DB location. Format is the libpq URI — works with `asyncpg`, `psycopg`, `node-pg`, and Alembic. |
| `STRIPE_WEBHOOK_SECRET` | Shared secret Stripe uses to sign webhook payloads (HMAC-SHA256). The ingest-edge verifies every webhook against this secret. `whsec_` prefix mimics Stripe's real format, so anyone reading the file sees it's a Stripe webhook secret, not an API key. |
| `STRIPE_API_KEY` | Needed by the Stripe SDK — even though we only *verify* webhooks (no outbound API calls yet), the SDK instantiation requires a key. Using `sk_test_` prefix makes "test mode" visible to anyone reading the file. |
| `ZUORA_WEBHOOK_SECRET` | Zuora signs callbacks with an HMAC (not their production approach — they actually use basic auth — but our edge assumes HMAC for cleaner symmetry with Stripe). Documented in README as a deliberate simplification. |
| `INGEST_PORT=3001` | Fastify listener port. Default 3001 (not 3000) to avoid conflict with common dev frontends. |
| `CORE_PORT=8000` | Uvicorn listener port. Default 8000 is FastAPI convention. |

### Alternatives considered
- **One URL with credentials baked in vs. separate host/port/user/pass vars.** Rejected split form: harder to pass to migration tools, every library takes the unified URL format. Split form is what you do if you have env-template ACL constraints (e.g. some fields safe to log, others not); we don't.
- **Use actual Stripe test-mode credentials checked in.** Rejected: even test-mode keys can be rate-limited or revoked by Stripe. Placeholder values force every developer to populate their own `.env`.
- **No `.env.example` at all, just README instructions.** Rejected: `.env.example` is copy-paste setup, which is what you want new contributors (and your future self) to hit.

### What's NOT in `.env.example` (deliberately)
- No `PYTHONPATH` — `pyproject.toml` handles package installation.
- No `NODE_ENV` — Fastify doesn't need it for M1; we'd add it if we introduced conditional logic that differs between dev and prod.
- No log-level var — defaults in each service are fine for M1; adding config knobs before we need them is YAGNI.

## File 3: `docker-compose.yml`

### What it contains
```yaml
services:
  postgres:
    image: postgres:16
    environment:
      POSTGRES_USER: finledger
      POSTGRES_PASSWORD: finledger
      POSTGRES_DB: finledger
    ports:
      - "5432:5432"
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U finledger"]
      interval: 2s
      timeout: 2s
      retries: 10
    volumes:
      - pgdata:/var/lib/postgresql/data

volumes:
  pgdata:
```

### Why each stanza

| Element | Reason |
|---------|--------|
| `image: postgres:16` | Pinned major version. `postgres:latest` is a trap — future minor versions can change default config (e.g., stricter SSL requirements) and break CI without warning. 16 is current LTS as of 2026. |
| `POSTGRES_USER/PASSWORD/DB = finledger` | Same value for all three = shortest possible config, and the creds are not secret (this is dev-only). Matches the `DATABASE_URL` in `.env.example`. |
| `ports: 5432:5432` | Host:container port mapping. We expose Postgres on the host so local Python/Node code (not running inside Docker) can connect via `localhost:5432`. In a production compose stack we'd not publish this port. |
| `healthcheck` | Docker Compose can wait for dependencies with `depends_on: condition: service_healthy`. Without a healthcheck, `docker compose up -d postgres` returns "started" while Postgres is still booting, and the next migration command fails. This healthcheck polls `pg_isready` every 2s up to 20s total — safe for a freshly-booted Postgres. |
| `volumes: pgdata:...` | Named volume for Postgres data. Without it, every `docker compose down -v` wipes the DB. With it, dev data persists across restarts. The volume is defined at the bottom of the file under the top-level `volumes:` key. |

### Alternatives considered
- **`postgres:16-alpine`**. Alpine is smaller but historically has had locale and glibc issues with some Postgres extensions. We'll want `pgcrypto` for `gen_random_uuid()` — the default `postgres:16` image has it preinstalled, alpine sometimes doesn't ship contrib modules. Not worth the risk for a portfolio project.
- **Bind mount instead of named volume** (`./pgdata:/var/lib/postgresql/data`). Rejected: Postgres's data dir on Windows with bind mounts has permission headaches (UID mismatches). Named volumes Just Work on all platforms.
- **Add ingest-edge and core as compose services right now.** Rejected: M1 development is faster when the Python and Node code run natively on the host (hot reload, debugger attach, native IDE integration). Postgres is the only thing worth containerizing for M1.

## File 4: `README.md` (stub)

The full README lands in Task 29. For Task 0 we write a minimal "here's how to run it" that a fresh contributor could use on day one. It will be overwritten at the end of M1 — acceptable because the stub has no content we don't want to keep (no outdated claims, just run instructions).

### Why write a stub at all
- First impression when someone `git clone`s: a blank repo looks abandoned, a stub with run instructions looks alive.
- Gives Task 30's manual demo a place to land.
- Forces us to put "how to run" in one place from the start, not scattered across six files.

## What Task 0 does NOT do

- **No Python project setup** — that's Task 1. We want the migration-less empty DB to stand alone first.
- **No Node project setup** — Task 18.
- **No CI config** — Task 28.
- **No migrations** — Task 2 onward.

Scoping Task 0 tight = first green commit is small, easy to verify, unambiguously complete.
